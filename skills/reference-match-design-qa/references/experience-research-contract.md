# Experience Research Contract

Research date: 2026-05-18

Purpose: force design decisions to be backed by user need, product evidence,
or an explicit assumption. Use this for apps, platforms, dashboards,
onboarding, navigation changes, repeated workflows, and any trust-critical flow.

Good design is not only visual taste. It is also choosing the right problem,
the right mental model, and the right proof that the design helped.

## Research Method Selection

| Situation | Use |
| --- | --- |
| The user goal or workflow is unclear | short interview, support-ticket scan, existing analytics, stakeholder note |
| Navigation or labels are changing | card sorting, tree testing, search query review, top-task analysis |
| A flow looks good but may be confusing | moderated usability test, first-click test, task walkthrough |
| The product is live and measurable | funnel review, session replay review, event analytics |
| The product is new and no data exists | explicit assumption log plus fastest validation plan |

## Experience Rules

- Name the top task before choosing the layout.
- Use language the user would use, not internal team language.
- Make status, next step, and recovery path visible.
- Reduce memory load: keep important choices and entered information visible
  when users need them later in the flow.
- If research evidence is unavailable, say so and treat the design decision as
  an assumption to verify, not as proven truth.
