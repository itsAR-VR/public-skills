# Negation Bias

When revising after feedback, do not keep the criticized idea in the draft and negate it. If the user says a framing is wrong, the fix is usually removal, not "not X, but Y."

## Rule

- If a piece should not be in the draft, remove it.
- If a piece was over-emphasized, reduce its footprint and move it behind the true main point.
- Write what the thing is, not what it is not.
- Do not lead with a rejected frame just to knock it down.

## Common Failure Shapes

- "This is not X; it is Y."
- "Rather than X, we do Y."
- "Unlike the old framing..."
- "To be clear, this is not..."
- Opening a revised paragraph with the thing the user asked to remove.

## Legitimate Exceptions

Keep negation when it is the actual substance:

- Strength or scope limits: "This is a preference, not an absolute rule."
- Safety warnings and correctness constraints.
- Worked examples that explicitly compare wrong and right.
- Real decision history where a rejected option must be recorded.

## Recovery When Invoked

If the user types `/neg`, rewrite the affected passage from the positive truth. Do not simply delete a few negative words; rebuild the surrounding paragraph so the wrong frame is gone.
